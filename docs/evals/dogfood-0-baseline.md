# Dogfood 0 — readability & chunk-quality baseline

**Date:** 2026-07-16 · **Slice:** #8 of `docs/spec/00-chunker-and-naive-book.md` · **Traces:** R-034 (chunk-quality eval loop), R-002 (low-signal), R-001 (coverage), R-025 (repo-agnostic)

This is the first end-to-end use of code-story on a real merged PR, and the numbers below are
the baseline every future chunking/ordering change gets compared against. The review was done
two ways: a full keyboard-only walk of the book UI (Playwright-driven, latencies sampled), and
a human-style read of the markdown export from cover to cover.

## Subject

[lexbox PR #2357](https://github.com/sillsdev/languageforge-lexbox/pull/2357) — "Add activity
filters to the activity view" (`277e418d8~1..277e418d8`). 30 files, +1622/−156. A genuinely
mixed diff: C# service + endpoint + 15 new tests, Svelte components, a pure-TS utils module,
9 Reinforced.Typings-generated type files, 2 UI-library tweaks, 8 `.po` locale catalogs.

Commands: `code-story "277e418d8~1..277e418d8"` (book UI) and `--export` / `--dump-chunks`
(baseline data). Compile time was a few seconds; nothing to record there.

### Signal accounting — the headline finding

Roughly **33 of 125 chunks (~26%) were noise the reviewer had to Enter through**:

- 12 chunks from `generated-types/**` — the classifier missed them twice over: the marker
  regex wants `code generated by` but Reinforced.Typings writes "This code **was** generated
  by", and the path pattern only matches a segment literally named `generated`
  ([#9](https://github.com/myieye/code-story/issues/9)).
- 16 chunks from 8 `.po` locale catalogs — the same ~15 strings repeated once per language
  ([#10](https://github.com/myieye/code-story/issues/10)).
- ~5 whitespace-only chunks (added blank lines between methods that got their own chunks),
  including one that is literally `+2 -0` of blank lines
  ([#11](https://github.com/myieye/code-story/issues/11)).

So the *mechanism* built in #7 works (verified there on a lockfile diff), but the *detector*
was tuned to one diff shape. Dogfood 0's diff shape found its edges immediately — which is
exactly what dogfooding is for.

## Readability read-through (markdown export)

**What worked well — better than expected:**

- Symbol-level chunking of C# is genuinely pleasant to review. Each of the 15 test methods is
  its own titled chunk (`HistoryServiceActivityTests.ProjectActivity_FiltersByAuthorId`), so
  the test file reads like a table of contents of behaviors. Same for `HistoryService`
  methods. This granularity is the product working as intended.
- The Svelte regex split (script/template) held up on real components: script fragments and
  `markup-region` chunks were coherent units.
- Fenced diffs render cleanly; the `kind · +adds -dels` meta line is enough context to
  calibrate attention before reading.
- Coverage means zero "wait, did I see that hunk?" anxiety — the queue is the review.

**What hurt — in order of pain:**

1. **Dependency-inverted ordering** ([#13](https://github.com/myieye/code-story/issues/13)):
   `ActivityFilter.svelte` consumes 8 functions from `activity/utils.ts`, but utils.ts comes
   ~500 export lines later; the tests come *before* the service they test; generated types and
   UI-library tweaks interleave mid-story. Naive file order is the deliberate M0 design — this
   is the R-034 evidence for the M1 ordering pass.
2. **Noise chunks** (the 26% above).
3. **Mid-statement fragment cuts** ([#12](https://github.com/myieye/code-story/issues/12)):
   `ApplyActivityFilters.fragment 2` opens with an orphaned `);` — the 40-line cap slices raw.
4. Tiny-chunk fragmentation: 25% of chunks are ≤3 lines. Fine for skimming, but each costs a
   full stop in the walk; merging trivial siblings is worth an experiment alongside #11.

**Review value check:** reading the book surfaced real review-worthy observations (e.g.
`HistoryView` passing `changeTypes: []` into `ActivityItem`; the double-meaning of an empty
`AuthorFilterKeys` array vs `null`) — the format supports actual code review, not just
coverage bookkeeping.

## Baseline verdict

Coverage, latency, and keyboard flow are solid; C# symbol granularity is the standout. The two
things that decide whether this tool beats a wall of diffs — **signal ratio** and **story
order** — both have concrete, measured gaps now: 26% noise chunks and 2 major ordering
inversions on a representative PR. Those numbers are the bar for the next iteration:
noise chunks → ~0 via #9/#10/#11, ordering inversions → 0 via the M1 ordering pass (#13).
