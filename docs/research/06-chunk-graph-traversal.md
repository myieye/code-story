# Research Report 06 — The Chunk Relationship Graph & Lawn-Mower Traversal (R-048 / R-049)

*Compiled 2026-07-17. Extends [`docs/research/01-landscape.md`](../research/01-landscape.md) and [`docs/research/04-review-science-and-leads.md`](../research/04-review-science-and-leads.md) — read those first; this report does not repeat their citations except where directly load-bearing. Every external claim carries a URL; internal claims carry file paths. Framing rule (Tim): absence of prior art is not evidence against the idea — for each comparison the question is "what did they build, what did they learn, does their evidence bear on THIS design," never "nobody does it, therefore risky."*

**The idea under test (R-048/R-049, [`inventory.md`](../requirements/inventory.md)):** beneath the linear book "backbone," a graph where each chunk knows its related chunks, *which lines* create each relationship, and each link visibly carries the linked chunk's review state (reviewed / next-step-only / unexplored-behind-it). A chunk *can* be marked reviewed before its neighbors are. The reviewer criss-crosses the graph like a lawn-mower robot; the system still guarantees 100% coverage (R-001), and small independently-markable chunks minimize ever re-reviewing the same chunk twice.

---

## 1. Review-state-per-unit tools — what unit, what invalidates, what users say

The "minimize re-review" half of R-049 is the most heavily prior-arted part of the idea. Every mainstream tool tracks review state, but **all at file granularity, and all keyed to a revision/patchset/iteration boundary** — never to a sub-file chunk with an identity that survives independently of the file.

- **Reviewable.io** — the strongest shipped semantics. It tracks review as a **per-file × per-revision × per-reviewer** matrix: "who reviewed which revision of each file" ([review model](https://docs.reviewable.io/reviews.html), [files](https://docs.reviewable.io/files.html)). A new revision re-flags only what changed; the "review chip" shows the file's overall state at latest revision while your per-file button shows *your* state at your current diff bounds — the two can legitimately disagree ([tracking changes](https://www.reviewable.io/blog/tracking-changes-in-a-code-review/)). Deferring a discussion parks it until a new revision or comment reactivates it ([completion conditions](https://www.reviewable.io/blog/completion-conditions-and-obsolete-revisions/)). **Bearing on our design:** Reviewable already proves the *value* of "don't re-review what didn't change" and the *disagreement* between personal and global state — but it invalidates at the whole-file/revision grain. Our chunk-identity-by-fingerprint model ([`core-primitives-sketch.md`](../design/core-primitives-sketch.md) §1, §6: `reopened` set when identity matching says the chunk materially changed) is Reviewable's matrix *one level deeper*, which is exactly the differentiator §6(a) of `01-landscape.md` already claimed. R-049's "minimize re-review" is Reviewable's core promise made sub-file.

- **Gerrit** — per-file reviewed checkbox, optionally auto-marked on view. Scores and (historically) reviewed-file flags **reset on a new patchset** unless `copyAllScoresIfNoCodeChange` applies (same parent tree + same delta) ([gerrit review cmd](https://gerrit-review.googlesource.com/Documentation/cmd-review.html), [reset discussion](https://groups.google.com/g/repo-discuss/c/gQahYXAYxB8)). Coarse and revision-scoped; the "same code delta ⇒ carry forward" rule is a crude file-level ancestor of our fingerprint-based carry-forward.

- **GitHub "Viewed" checkbox** — file-level; **auto-unchecks when that file changes in a later commit**, adding a "Changed since last view" tag ([Interactive Logic](https://www.interactivelogic.net/using-github-viewed-checkbox/), [reset guide](https://codewithhugo.com/github-review-reset-viewed/)). Documented pain: no bulk reset ([community #133275](https://github.com/orgs/community/discussions/133275)), no per-commit granularity ([community #5303](https://github.com/orgs/community/discussions/5303)), and collapsed generated files still pollute the Viewed tracker ([community #186824](https://github.com/orgs/community/discussions/186824), noted in `04` A2). This is the crude baseline R-049 improves on: file-grain, all-or-nothing, resets on any change to the file even if your chunk was untouched.

- **Azure DevOps iterations** — each push is a numbered **iteration**; reviewers diff iteration-to-iteration and focus only on new changes, and CI posts status per iteration ([review PRs](https://learn.microsoft.com/en-us/azure/devops/repos/git/review-pull-requests?view=azure-devops), [iteration changes API](https://learn.microsoft.com/en-us/rest/api/azure/devops/git/pull-request-iteration-changes/get?view=azure-devops-rest-7.1)). Praised as more robust than GitHub's "changes since last review" ([2026 guide](https://dev.to/rahulxsingh/azure-devops-code-review-tools-and-setup-guide-2026-5b5i)). Still file/iteration-grained.

**Section verdict:** the "don't re-review unchanged work" mechanic is validated and wanted (Reviewable exists and users defend it; McIntosh MSR'14 in `04` A3 ties coverage to quality). No tool tracks it at sub-file chunk grain with identity that survives file edits. That part of R-048/R-049 is a real, evidence-backed refinement, not a novelty gamble. The genuinely new claim is not "track review state" but "**track it per chunk and let the graph show a neighbor's state**."

## 2. Graph / visual code review

- **CodeSee Review Maps** — the closest shipped precedent to R-048's visual change graph: auto-diagrams of how a PR's changed files relate, auto-updated on merge ([review map guide](https://docs.codesee.io/docs/review-map-guide), [code reviews](https://www.codesee.io/code-reviews)). **It is dead as a standalone product** — shut down Feb 2024, acquired by GitKraken May 2024, the dedicated platform sunset ([GitKraken/CodeSee](https://www.gitkraken.com/blog/gitkraken-launches-devex-platform-acquires-codesee), [PRNewswire](https://www.prnewswire.com/news-releases/gitkraken-acquires-codesee-launches-devex-platform-including-support-for-google-geminis-ai-model-302144298.html)); a successor, NanoAPI, pitched itself as "carrying the torch" ([HN](https://news.ycombinator.com/item?id=42114034)). I could **not** retrieve direct HN post-mortem quotes on *why* Review Maps didn't stick (thread returned HTTP 429; general search surfaced no specific "developers didn't use it" testimony — **reception evidence: not found directly**). What the general software-visualization literature says instead is in §5(c). `01-landscape.md` already draws the lesson: "visualization alone, without workflow integration, didn't survive." That is the most important cited datum for R-048 and it is a caution, not a veto — the map was a *separate artifact beside* the diff, not the reading surface itself.

- **CodeRabbit Code Graph Analysis** — builds a **fresh AST-based structural dependency graph of the repo per review** (not pre-indexed), to find cross-file downstream breakage, and uses it to cluster changed code into cohorts and order them ([changelog](https://docs.coderabbit.ai/changelog/code-graph-analysis), [context engineering](https://www.coderabbit.ai/blog/context-engineering-ai-code-reviews)). This is the strongest evidence that **per-review chunk-relationship graphs are computationally feasible at product scale** — but the graph feeds *the bot's* comments and cohort ordering; it is never exposed to the human as a navigable, status-annotated surface.

- **Greptile** — pre-builds a whole-repo graph of files/functions/variables/classes and queries it per file to know imports, callers, callees, usage sites; ships auto-generated sequence diagrams of call flows ([graph-based context](https://www.greptile.com/docs/how-greptile-works/graph-based-codebase-context), [agent](https://www.greptile.com/agent)). Again: graph is internal to the bot (prior art for our context payloads / import-graph, not for reviewer traversal).

- **Baz.ai Change Request Graph** — the single closest shipped thing to R-048's *visible* graph: "a visual graph showing relationships between code elements (added / modified / removed), helping reviewers understand how changes propagate," embedded in the review workspace alongside diff, topics, and description ([docs](https://docs.baz.co/code-review/web), [Bedrock writeup](https://aws.amazon.com/blogs/machine-learning/how-baz-improved-its-ai-agent-code-review-accuracy-using-amazon-bedrock-agentcore/)). **What it is NOT** (per `01-landscape.md` §2 and docs): the nodes are code elements, not review *chunks*; there is no per-chunk reviewed-state carried on the *links*; progress is commit-granular, not chunk-granular; edges are not documented as line-provenanced; and it is closed/cloud. Baz proves the *visual change graph* is a shippable review feature; it does not occupy the R-048 combination.

- **Graphite / stacks** — PR-level dependency graphs (author-side stacking). Relevant UX evidence only: reviewers accept graph-of-units navigation when each unit is small and independently mergeable — the same "small independently-markable units" bet R-049 makes, one grain up. `01-landscape.md` covers this.

- **CodeScene Delta Analysis** — not a graph, but the mature precedent for **status/risk-aware prioritization steering where the reviewer spends effort**: classifies per-PR delivery risk relative to the repo, flags hotspot-touching PRs, recommends a review level, explicitly to "focus your time where it's likely to be needed most" and combat reviewer fatigue ([delta analysis docs](https://docs.enterprise.codescene.io/versions/4.5.0/guides/delta/automated-delta-analyses.html)). Bears on R-048's "next-step-only vs unexplored-behind-it" status coloring: prioritization signals on a review surface are validated and wanted.

**Section verdict:** graphs are everywhere in 2025-26 review tooling, but as **bot-internal reasoning substrates** (CodeRabbit, Greptile) or **standalone visual artifacts** (CodeSee dead; Baz alive but element-grained, cloud, no per-link review state). Nobody exposes a *chunk* graph whose *links carry review state* as the reviewer's traversal surface.

## 3. Academic / industrial research

The research literature is where R-048/R-049 finds its strongest *positive* support — the individual ingredients are separately validated, and one ingredient (sub-statement dependency graphs for change comprehension) is almost exactly R-048.

- **Call-graph navigation beats linear/list navigation — measured.** Stacksplorer and **Blaze** (call-graph exploration IDE tools) **significantly increased success rates and significantly reduced task-completion time** on maintenance tasks vs Eclipse's Call Hierarchy and vs unmodified Xcode; notably the Call Hierarchy exposes *more* of the graph yet did *not* reduce times — locality/relevance of the shown neighborhood mattered more than completeness ([Stacksplorer UIST'11 PDF](http://people.eecs.berkeley.edu/~bjoern/papers/karrer-stacksplorer-uist11.pdf), [RWTH page](https://hci.rwth-aachen.de/stacksplorer)). **Prodet** deployed an always-on filtered call-graph view in a **six-week professional field study** and found a *statistically significant, sustained increase* in structural navigation ([Prodet ICSE'15](https://ieeexplore.ieee.org/document/7202967/), [PDF](https://www.semanticscholar.org/paper/A-Field-Study-on-Fostering-Structural-Navigation-Augustine-Francis/dbc9cba16341dfe1f99d7167a5f5f4a046b9240d)). **Bearing on R-048/R-049:** this is direct evidence that letting people traverse a *filtered, locally-relevant* structural graph improves comprehension outcomes and gets adopted — the mechanism R-049 proposes. The Call-Hierarchy null result is the key design lesson: **show the relevant neighborhood, not the whole graph.**

- **Optimal reading order = grouping by relatedness (Baum & Schneider).** Their thesis and the CoRT tool conclude "an optimal order is mainly an optimal grouping of the change parts by relatedness," derived from interviews + survey ([Cognitive-support code review tools thesis](https://repo.uni-hannover.de/handle/123456789/9217), [CoRT repo](https://github.com/tobiasbaum/reviewtool), [tobiasbaum.github.io](http://tobiasbaum.github.io/)). This is the theoretical bridge between the linear book (R-005) and the graph (R-048): the *ordering* is a linearization of the *relatedness graph*. R-048 says: keep the graph, don't only ship its flattening.

- **Salient-class + Code-Change Dependency Graph — R-048 in academic form.** Huang et al.'s **ISC** identifies the "salient class" that drives a commit so reviewers understand it first; **GBSCI** improves this by building **sub-statement-level program-dependency graphs before/after the change, merging them into a Code-Change Dependency Graph, and learning over it with R-GCN** ([Assessing LMs for Salient Class Identification, arXiv 2606.21629](https://arxiv.org/pdf/2606.21629)). A *sub-statement dependency graph of a change* is precisely R-048's "which lines are responsible for the relationship." **This is the closest thing to a peer-reviewed validation that line-provenanced change-dependency graphs improve reviewer comprehension** — the research community built exactly this data structure for exactly this purpose. It has not been turned into a status-annotated reviewer traversal UI; that is the open slot.

- **Change decomposition improves *understanding* (not just defect count).** ClusterChanges (>40% of changesets decomposable by def-use), di Biase's controlled experiment (decomposition positively associated with reviewer understanding), SmartCommit/UTANGO hunk-graph partitioning — all catalogued in `04` A1. R-048's chunks-with-edges is the persistent, navigable form of that same def-use partition.

- **Graph-structured navigation vs retrieval (2026).** "The Navigation Paradox in Large-Context Agentic Coding" reports graph-structured dependency navigation *outperforms* flat retrieval on architecture-heavy tasks ([arXiv 2602.20048](https://arxiv.org/html/2602.20048v1)). Bears on the "graph beats a flat ordered list when structure matters" premise, in the agent-navigation setting.

- **Position bias / no-universal-order** (`04` A1: Fregnan FSE'22, Bagirov EASE'23, Olewicki FSE'24 +23% comments from smart reorder, "Breaking the Alphabet" ICSE'26 10.2%, "Not One to Rule Them All" arXiv 2506.10654). These support *any* non-alphabetical structuring — including a graph — and specifically support R-049's premise that **no single linear order is optimal for everyone**, which is the argument for *also* allowing free traversal.

**Section verdict:** the research is net-*positive* toward R-048/R-049, more than the product landscape suggests. Graph traversal improves comprehension and is adopted (Prodet field study); relatedness-grouping is the theory of good order; sub-statement change-dependency graphs are a published comprehension aid (GBSCI). The recurring caveat — **filtered/local neighborhood, not the whole graph** (Stacksplorer's Call-Hierarchy null) — is the single most important design input.

## 4. Adjacent products — does ANY give per-chunk review state on a navigable chunk graph with line-level edge provenance?

**Direct answer: No.** The scorecard on the R-048/R-049 combination:

| Tool | Chunk (sub-file) grain? | Graph exposed to human? | Edges line-provenanced? | Per-unit review state on links? | Free traversal + guaranteed coverage? | Local / BYO-agent? |
|---|---|---|---|---|---|---|
| Reviewable | file | no | n/a | file×revision (not on links) | no coverage-queue | no |
| GitHub / Gerrit / ADO | file | no | n/a | file/iteration | no | no |
| CodeRabbit (Code Graph) | cohort/hunk (internal) | no (bot uses it) | AST edges (internal) | no | no | no |
| Greptile | function (internal) | sequence diagrams only | internal | no | no | no |
| **Baz** | code element | **yes (change graph)** | not documented | commit-granular, not on links | no | no |
| CodeSee (dead) | file/symbol | yes (map) | call/import edges | no | no | no |
| cubic / Stage / Haystack / Devin | file-group / hunk | no (ordered walkthrough) | no | finding- or chapter-level | no | no |

Closest partials, precisely bounded:
- **Baz** exposes a visual change graph (§2) — but element-grained, no per-link review state, edges not line-provenanced, cloud.
- **Reviewable** has the deepest review-state model — but file×revision, no graph.
- **CodeRabbit/Greptile** have the richest graphs — but internal to the bot.
- **Devin/cubic/Stage/Haystack** (from `01-landscape.md` §2) linearize a relatedness graph into an ordered walkthrough and *discard the graph* — they ship R-048's flattening without R-048.

The combination **{chunk grain + line-provenance edges + status-aware links + free traversal + guaranteed coverage + linear backbone}** is **unoccupied**, consistent with `01-landscape.md`'s headline verdict for the broader vision.

## 5. The hard questions for THIS design (analysis, not a list)

**(a) Does free traversal undermine the comprehension benefit of a curated linear order?** This is the sharpest internal tension. `04` A1 is unambiguous that **understanding is the bottleneck** (Bacchelli & Bird ICSE'13) and that **grouping-by-relatedness order** aids it (Baum & Schneider §3). A pure lawn-mower criss-cross risks fragmenting the narrative context the book was built to provide. *But* two facts cut the other way: (1) "no single order is optimal" and 44.6% of PRs are already reviewed out of order (`04` A1, arXiv 2506.10654) — reviewers criss-cross *anyway*, today, without support; (2) the call-graph studies (§3) show *guided local* traversal helps. **Resolution the design already encodes:** R-048 keeps the linear book as the **backbone/default path** (Tim: "a really good backbone") and the graph as an *optional overlay*. The risk is real only if the graph becomes the *primary* surface and the backbone atrophies. Design gate: the book order must remain the thing a reviewer can follow mindlessly to 100%, with the graph as an affordance for the reviewer who *chooses* to jump — never the default.

**(b) Does per-chunk "reviewed" + visible neighbor status create false confidence — where does compositional correctness get reviewed?** This is the strongest *genuine* risk, and it is documented: bugs "exist in how code lines interact with lines you didn't include," and reviewing a slice in isolation misses interaction with code outside the window ([dev.to](https://dev.to/leena_malhotra/what-broke-when-i-let-ai-handle-my-code-reviews-and-how-i-fixed-it-25c0), [SoftwareSeni](https://www.softwareseni.com/what-ai-code-review-catches-and-misses/)). A chunk can be individually correct and wrong in composition. Marking chunks green one-by-one can manufacture exactly the "the bot would have caught it" complacency `04` A5/D#6 warns about (Tufano et al. anchoring; arXiv 2412.18531 "if any other issue existed…"). **R-048 partially answers this itself:** the line-provenance edges *are* the composition surface — an edge says "these lines here depend on that chunk there," so composition is reviewable *as edges*, not only as nodes. But that only works if the design treats **"all touching edges seen"** as a distinct, surfaced state from "node reviewed." Recommendation for the spec: coverage (R-001) must be defined over **edges as well as chunks** — a green node with an unexamined edge to a red node is precisely the "next-step-only, but not everything behind it" state R-048 names, and it should read as *incomplete*, not done. This is the design's answer to false confidence, and it must be built in, not assumed.

**(c) UI-complexity budget — graph UIs fail from overwhelm.** The literature is blunt: "when cognitive load becomes too high due to excessive features, users simply stop using visualization tools"; large graphs drive cognitive load; feature-creep reduces adoption ([Frontiers systematic review](https://www.frontiersin.org/journals/communication/articles/10.3389/fcomm.2025.1605655/full), [graph-viz cognitive-load study](https://nschwartz.yourweb.csuchico.edu/huang%20eades%20&%20hong%202009.pdf)). CodeSee's death as a standalone (§2) and `01-landscape.md`'s "visualization alone didn't survive" are the cautionary data points (direct CodeSee user testimony: **not found**). The Stacksplorer Call-Hierarchy null result (§3) is the constructive counterpart: **the whole graph hurts; the filtered local neighborhood helps.** Design implication: R-048 should almost never render a whole-PR node-link diagram. The winning form is likely *ambient, local, in-flow* — the current chunk's immediate neighbors with their status, inline in the book, à la Prodet's always-on filtered view — not a separate canvas the reviewer must pan. This keeps the complexity budget with the backbone.

**(d) Does "minimize re-review" conflict with intentional re-reading?** Yes, and the repo already flagged it: **R-004 occurrences exist precisely so a chunk can appear in multiple contexts on purpose** ([`core-primitives-sketch.md`](../design/core-primitives-sketch.md) §2; occurrence-keyed rows are already built per CLAUDE.md's maintainability pass — walk stops = occurrences, progress = distinct chunks). R-049's "radically minimize re-reviewing the same chunk twice" is in *tension* with R-004's "show it again in a new context." The reconciliation is the distinction already in the model: **re-*review* (re-establishing correctness) is what we minimize; re-*encounter* (seeing a reviewed chunk as context for a new chunk) is valuable and stays.** The graph makes this cleaner than the linear book can: a neighbor shown as "already reviewed ✓" is a *context re-encounter* that costs the reviewer nothing (they see its status, don't re-audit it), which is exactly R-048's "clearly visible whether the linked chunk is already reviewed." So R-004 and R-049 are compatible *if* the UI distinguishes "re-audit" from "glance at a green neighbor." That distinction must be explicit in the spec.

**(e) Edge quality — line-provenance edges are only as good as reference extraction.** This is a hard feasibility ceiling, and the numbers are sobering. Static call-graph construction has **median recall ~0.884** even in mature analyzers, with dynamic dispatch, reflection, and native/indirect calls as the standard unsoundness sources ([On the Recall of Static Call Graphs, ICSE'20](https://dl.acm.org/doi/10.1145/3377811.3380441); [Total Recall? ISSTA'24](https://www.opal-project.de/articles/TotalRecall@ISSTA24.pdf)); adding precision doesn't help recall and vice-versa (they are separate failures). The repo's *current* edge machinery is far below even that bar **by deliberate design**: [`import-graph.ts`](../../packages/core/src/import-graph.ts) resolves only *changed-file-to-changed-file* import/using edges with "precision over recall, never fabricate an edge" (unresolved edges silently dropped); [`references.ts`](../../packages/server/src/references.ts) does a tree-sitter pass for callees/constructors/JSX components with **no scope analysis at all** (locals and params are left in; the resolver drops what it can't place) and a hand-maintained stoplist. **Bearing on R-048:** the edges will be *incomplete* (many real relationships missing) and *sometimes wrong* (tail-identifier matching can conflate `foo.bar()` and `baz.bar()`). That is survivable *only* because of the design's own escape hatch: the linear book + R-001 coverage guarantee means **a missing edge costs the reviewer a jump affordance, never coverage** — the lawn-mower still mows every chunk via the backbone even if the graph is Swiss cheese. A *wrong* edge is worse than a missing one (it sends the reviewer to an irrelevant chunk and pollutes the status display), so the current "precision over recall" stance in `import-graph.ts` is exactly right and must extend to reference edges. **Do not promise edge completeness; promise coverage, and present edges as best-effort hints.**

## SYNTHESIS — honest verdict

**Is the combination genuinely unoccupied?** Yes. §4's scorecard is conclusive: no tool gives per-*chunk* review state on a navigable chunk graph with *line-provenance* edges and *status-carrying links*, under a coverage guarantee, with a linear backbone. Baz ships the closest *visual change graph* but element-grained, cloud, no per-link state; Reviewable ships the deepest *review-state model* but file×revision and no graph; CodeRabbit/Greptile have the richest *graphs* but keep them inside the bot; Devin/cubic/Stage/Haystack ship the graph's *flattening* (ordered walkthrough) and throw the graph away. The unoccupied ground is real and it is a *combination*, not a single missing feature — the same shape as `01-landscape.md`'s verdict for the wider vision.

**Strongest evidence FOR (adjacent successes, not novelty-worship):**
1. **Call-graph traversal measurably improves comprehension and gets adopted** — Stacksplorer/Blaze (faster + more successful maintenance), Prodet (sustained adoption in a 6-week professional field study). Traversing a *filtered local* structural graph is a validated mechanic, and comprehension is the documented bottleneck (`04` A1).
2. **Sub-statement change-dependency graphs are a published reviewer-comprehension aid** (GBSCI / salient-class, arXiv 2606.21629). R-048's core data structure exists in the literature *for this exact purpose*; it has simply never been turned into a status-annotated reviewer traversal surface.
3. **"Minimize re-review at sub-unit grain" is wanted** — Reviewable's whole business is the file-grain version and users defend it; our chunk-identity model ([`core-primitives-sketch.md`](../design/core-primitives-sketch.md) §6) is that promise made finer.
4. **Reviewers already criss-cross** (44.6% non-alphabetical; no universal order) — R-049 *supports* existing behavior rather than imposing a new one.

**Strongest GENUINE risk (not novelty-as-risk):** *false confidence from node-by-node green-marking* (§5b), amplified by documented AI-review anchoring (`04` A5). Marking chunks reviewed one at a time, with neighbors shown green, is a near-perfect generator of "it's all fine, the pieces all passed" — while composition/integration bugs live exactly in the edges between pieces. The mitigation is intrinsic to R-048 *if built deliberately*: **make coverage a property of edges as well as nodes**, so "node reviewed but touching an unexamined/red neighbor" surfaces as *incomplete* ("next-step-only, not everything behind it" — R-048's own words), never as done. If the design lets a reviewer reach 100% while touching-edges go unexamined, it has automated complacency. Secondary risk: **UI overwhelm** (§5c) — mitigated by rendering the local neighborhood in-flow (Prodet-style), never a whole-PR canvas (CodeSee's grave). Tertiary constraint: **edge quality is capped by reference-extraction precision** (§5e, static call graphs ~0.88 recall; our machinery deliberately far below full recall) — survivable *only* because the coverage guarantee rides the linear backbone, not the graph; edges are hints, never load-bearing for coverage.

**Cheapest falsifiable dogfood experiment to validate the core bet.** The core bet is: *"a status-aware neighbor graph lets a reviewer criss-cross a real agent PR and reach 100% coverage with less re-reading and no worse comprehension than the linear book alone."* Falsify it cheaply, reusing what exists, no graph UI required first:
1. On an existing dogfood subject (lexbox PR 2357 / 2379 / 2309 — 125+ chunks, real edges), compute the chunk graph from the *already-built* [`import-graph.ts`](../../packages/core/src/import-graph.ts) + [`references.ts`](../../packages/server/src/references.ts) edges. **Measure edge precision by hand on a sample** (the R-048 make-or-break number): what fraction of proposed line-provenance edges point at a genuinely relevant chunk? If precision is low, R-048 is blocked on §5e before any UI matters.
2. **Instrument the existing keyboard walk** (`tools/dogfood-walk.mjs`, per CLAUDE.md) to log, per chunk marked reviewed, how many of its graph-neighbors were *already* reviewed (would-be "free glances") vs unreviewed-behind-it. This quantifies R-049's "minimize re-review" upside *from the current linear walk* — no new UI — and tells you whether the graph would actually save encounters or just decorate.
3. Only if (1) precision is acceptable *and* (2) shows meaningful would-be-saved re-encounters, build the smallest possible in-flow neighbor strip (current chunk's neighbors + their review-state color) and A/B it against the plain book on time-to-100%-coverage and a post-hoc comprehension probe (can the reviewer describe the cross-chunk interactions?). This is the only step that costs UI work, and steps 1-2 gate it.

The bet is worth testing and the ground is genuinely open. The one thing that would make it a *bad* build is shipping node-by-node green-marking without edge-level coverage — that converts the idea's headline feature into an anchoring/complacency machine. Build the edges as coverage, keep the linear book as the backbone that carries the R-001 guarantee, render locally not globally, and treat edges as best-effort hints — and the risks are all mitigable with mechanisms the repo has already partly built.
