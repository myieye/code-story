## Part A — What research actually says about reading and reviewing diffs

### A1. Diff reading order, changeset decomposition, and review size

**Understanding is the bottleneck, not commenting.** The foundational study of modern (change-based, tool-assisted) code review is Bacchelli & Bird, ["Expectations, Outcomes, and Challenges of Modern Code Review" (ICSE 2013)](https://sback.it/publications/icse2013.pdf) ([Microsoft Research page](https://www.microsoft.com/en-us/research/publication/expectations-outcomes-and-challenges-of-modern-code-review/)). From interviews, observation, and classification of review comments at Microsoft: finding defects is the *stated* top motivation, but actual outcomes skew toward knowledge transfer, awareness, and alternative solutions — and **"understanding the code and the reason for the change" is the single biggest challenge reviewers report; understanding consumes most review time, and most comprehension needs are unmet by review tools.** This is the strongest single academic mandate for a comprehension-first review product.

**Follow-ups confirm it.** MacLeod et al., ["Code Reviewing in the Trenches" (IEEE Software 2017)](https://dl.acm.org/doi/abs/10.1109/MS.2017.265100500) (18 interviews + 911 survey responses at Microsoft) again ranks *comprehension of the change* among the top challenges, alongside time pressure and social friction. Kononenko et al. found review usefulness inversely related to change size, and that time-to-review is driven by patch size, quality, and rationale availability (survey summarized in [this literature review](https://arxiv.org/pdf/2107.10375)). Pascarella, Spadini, Palomba, Bruntink & Bacchelli, ["Information Needs in Contemporary Code Review" (CSCW 2018)](https://dl.acm.org/doi/10.1145/3274404) ([PDF](https://fpalomba.github.io/pdf/Conferencs/C36.pdf)) catalogued what reviewers actually ask for in ~900 review threads from OpenStack/Android/Qt: the recurring needs are **suitability of alternative solutions, correct understanding of intent/rationale, whether code is correct in context, and splitting/organizing the change** — i.e., narrative and context, not lint output.

**Changeset decomposition works.** Barnett, Bird, et al., ["Helping Developers Help Themselves: Automatic Decomposition of Code Review Changesets" (ICSE 2015)](https://www.microsoft.com/en-us/research/wp-content/uploads/2016/02/barnett2015hdh.pdf) built **ClusterChanges**, which partitions a changeset by def-use relations between diff regions. Findings: **>40% of Microsoft changesets could be decomposed into multiple independent partitions**, and developers found the partitioning helpful in review ([the morning paper summary](https://blog.acolyer.org/2015/06/26/helping-developers-help-themselves-automatic-decomposition-of-code-review-changes/)). Di Biase et al. ran the controlled experiment: ["The effects of change decomposition on code review" (PeerJ CS 2019)](https://pmc.ncbi.nlm.nih.gov/articles/PMC7924728/) — decomposition showed a **positive association with reviewers' level of understanding of the changeset** (defect-finding differences were not the headline; comprehension was).

**Commit untangling is an active line through 2026.** [SmartCommit (ESEC/FSE 2021)](https://dl.acm.org/doi/abs/10.1145/3468264.3468551) builds a diff-hunk graph and interactively partitions tangled changes into activity-oriented groups ([code](https://github.com/Symbolk/SmartCommitCore)). [UTANGO (FSE 2022)](https://dl.acm.org/doi/10.1145/3540250.3549171) does context-aware graph-based clustering of code changes. LLM-era successors: [Atomizer, an LLM multi-agent intent-driven commit untangler (arXiv 2601.01233)](https://arxiv.org/html/2601.01233), an [LLM-driven collaborative untangling model (arXiv 2507.16395)](https://arxiv.org/pdf/2507.16395), plus [AtomicCommitBench — can coding agents reconstruct commit histories from squashed patches? (arXiv 2607.03332)](https://arxiv.org/pdf/2607.03332) and a [benchmark framework for untangling tools](https://github.com/Thomsch/untangling-tools-benchmark). Takeaway: grouping a diff into intent-coherent, ordered chunks is both feasible and validated as comprehension-enhancing — and LLMs are the current best untanglers.

**File *order* measurably changes review outcomes (position bias is real).**
- Fregnan et al. found that **files shown earlier in a review receive more comments than files shown later, even controlling for confounders** — the alphabetical file list is silently rationing reviewer attention (["First Come First Served: The Impact of File Position on Code Review", ESEC/FSE 2022; summarized in the arXiv line of work below](https://arxiv.org/abs/2306.06956)).
- Bagirov et al., ["Assessing the Impact of File Ordering Strategies on Code Review" (EASE 2023)](https://arxiv.org/abs/2306.06956): on 51,566 IntelliJ IDEA reviews, comment density concentrates on files listed first; ordering files by diff-relevance beats alphabetical on all ranking metrics for Kotlin files.
- Olewicki et al., ["An Empirical Study on Code Review Activity Prediction and Its Impact in Practice" (FSE 2024)](https://arxiv.org/abs/2404.10703): in a 29-expert user study, **re-ordering changed files by predicted review-activity yielded +23% more review comments and better hot-spot precision/recall than alphanumeric order**.
- ["Breaking the Alphabet: Rethinking File Ordering in Code Review" (ICSE 2026)](https://conf.researchr.org/details/icse-2026/icse-2026-research-track/310/Breaking-the-Alphabet-Rethinking-File-Ordering-in-Code-Review): survey of 1,355 professional developers across 182 OSS projects — **only 10.2% consider alphabetical ordering optimal**.
- ["Not One to Rule Them All: Mining Meaningful Code Review Orders from GitHub" (arXiv 2506.10654)](https://arxiv.org/pdf/2506.10654): mined real reviewer navigation; there is no single universal best order — order should adapt to change type (feature vs refactor vs fix).

**Review size and session limits — the real sources.** The oft-quoted numbers come from the SmartBear/Cisco study (2,500 reviews, ~50 developers, 10 months), written up in ["Code Review at Cisco Systems" (Cohen, SmartBear, 2006)](https://static0.smartbear.co/support/media/resources/cc/book/code-review-cisco-case-study.pdf) and distilled in SmartBear's ["Best Practices for Peer Code Review"](https://smartbear.com/learn/code-review/best-practices-for-peer-code-review/) / ["11 Best Practices" whitepaper](https://static1.smartbear.co/support/media/resources/cc/11_best_practices_for_peer_code_review_redirected.pdf):
- Review **≤200–400 LOC at a time**; defect-detection ability drops sharply beyond ~400 LOC.
- Inspection rates **faster than ~500 LOC/hour show a significant drop in defect density found**; ~300 LOC/hour is a reasonable target.
- A 200–400 LOC review over **60–90 minutes** yields 70–90% defect discovery; **never review more than ~60–90 minutes in one sitting** (vigilance/fatigue decrement — consistent with classic inspection literature and general sustained-attention research; industry syntheses: [CodeAnt on mental drain in reviews](https://www.codeant.ai/blogs/mental-simulation-code-reviews), [Atomic Robot on AI review fatigue](https://atomicrobot.com/blog/ai-review-fatigue/)).
- Caveats: this is an industrial study, not a randomized experiment, and it predates modern tooling — but it remains the best-quantified source, and academic findings (size ↔ usefulness inversely related; large PRs → cognitive overload) point the same direction ([Bruno Arine's synthesis with further citations](https://brunoarine.com/blog/pull-requests-should-have-less-than-200-cl/)).

**Google's converged practice matches.** Sadowski et al., ["Modern Code Review: A Case Study at Google" (ICSE-SEIP 2018)](https://sback.it/publications/icse2018seip.pdf) (12 interviews, 44 survey, 9M reviewed changes): Google converged on **very small changes (median change is tiny), one reviewer usually sufficient, fast turnaround**, with explicit ownership/readability. Rigby & Bird's convergent practices (lightweight process; early/quick/frequent; **small changes**; ~2 reviewers; review as group problem-solving) are confirmed. Implication: when you *can't* make the change small (agent-generated PRs), tooling must simulate smallness — decompose, order, and pace.

**LLM-generated review guides / walkthroughs (2024–2026).** This is an emerging, directly relevant literature:
- ["Rethinking Code Review Workflows with LLM Assistance: An Empirical Study" (arXiv 2505.16339)](https://arxiv.org/abs/2505.16339) — field study + field experiment (WirelessCar): confirms reviewer pain is context-switching and missing context; **automatic summarization of complex PRs is one of the highest-value LLM assists; false positives and trust are the top concerns.**
- ["Help Me to Understand this Commit! A Vision for Contextualized Code Reviews" (arXiv 2402.09528)](https://arxiv.org/html/2402.09528) — argues review tools should assemble the *context* a reviewer needs per hunk (callers, history, rationale), essentially a research agenda for a review-guide product.
- ["Rethinking Code Review in the Age of AI: A Vision for Agentic Code Review" (arXiv 2605.17548)](https://arxiv.org/pdf/2605.17548) — proposes a five-stage agentic pipeline (PR creation → augmentation → reviewer selection → AI-assisted review → retrospective) with humans as "supervisory operators" holding judgment and accountability at explicit checkpoints; stresses aligning generation pace with *shared understanding*.
- ["These Aren't the Reviews You're Looking For: How Humans Review AI-Generated Pull Requests" (arXiv 2605.02273)](https://arxiv.org/pdf/2605.02273) — mined the AIDev dataset: **most AI-generated PRs receive no human review at all**; when reviewed, interaction is "agent steering" rather than standalone human evaluation. The human-oversight gap this product targets is empirically documented.
- [CodeReviewQA (arXiv 2503.16167)](https://arxiv.org/html/2503.16167) benchmarks whether LLMs *comprehend* review comments; [SWR-Bench (arXiv 2509.01494)](https://arxiv.org/html/2509.01494v1) benchmarks PR-level review with full project context; [survey of code-review benchmarks pre/post-LLM (arXiv 2602.13377)](https://arxiv.org/html/2602.13377v1).

### A2. "Diluted" UI/markup code vs "dense" backend code — what the evidence supports

Direct academic evidence comparing *review effort or defect density of frontend/markup vs backend code* is **thin; treat the diluted/dense framing as a plausible hypothesis, not established fact**. What exists:

- **Language/file-type effects on defect proneness are weak.** Ray et al.'s famous ["A Large Scale Study of Programming Languages and Code Quality in GitHub" (FSE 2014)](https://www.researchgate.net/publication/269634115_A_large_scale_study_of_programming_languages_and_code_quality_in_github) claimed language↔defect associations, but Berger et al.'s reproduction, ["On the Impact of Programming Languages on Code Quality" (TOPLAS 2019)](https://dl.acm.org/doi/10.1145/3340571), knocked it down: **only 4 of 11 associations survived, with "exceedingly small" effect sizes** (see also the [rebuttal-of-the-rebuttal exchange](https://arxiv.org/abs/1911.11894)). Lesson: don't hard-code "this file type is safe"; per-change signals beat per-type priors.
- **Some change categories demonstrably get less review attention.** Spadini et al., ["Test-Driven Code Review" / test-code review studies (ICSE 2019)](https://sback.it/publications/icse2019a.pdf) found **test code receives significantly less review discussion than production code**; a [2026 replication in the GitHub-Actions era (arXiv 2603.15935)](https://arxiv.org/abs/2603.15935) revisits this. So differential attention by file category is real reviewer behavior — the question is whether it's *calibrated*.
- **Industry practice already encodes "some diffs aren't worth human reading."** GitHub's `linguist-generated` attribute **collapses generated files (lockfiles, build output) by default in PR diffs** ([GitHub docs](https://docs.github.com/en/repositories/working-with-files/managing-files/customizing-how-changed-files-appear-on-github), [thoughtbot guide](https://thoughtbot.com/blog/github-diff-supression)); a [community discussion](https://github.com/orgs/community/discussions/186824) notes the pain that collapsed generated files still pollute the "Viewed" progress tracker. This is de-facto prior art for tiered review attention.
- **Defect-prediction literature says risk follows churn, history, and complexity — not file extension.** (See A4.) The defensible product framing: UI/markup hunks often carry *fewer decisions per line* (dilution), so a guide should compress them into higher-altitude summaries — but the tool should verify low-risk status per hunk (no logic, no security surface, matches design tokens) rather than assume it from file type.

### A3. Checklists, perspective-based reading, and coverage tracking

- **Perspective-Based Reading (PBR)** — Basili et al., ["The Empirical Investigation of Perspective-Based Reading" (Empirical SE 1996)](https://www.cs.umd.edu/~mvz/handouts/emp_pbr.pdf): reviewers each read from an assigned perspective (tester, designer, user) with an operational scenario. In NASA SEL experiments PBR teams achieved better coverage/detection than ad-hoc reading, largely via **low overlap between perspectives** — though a [replication](https://link.springer.com/article/10.1007/s10664-006-5967-6) found the advantage disappears when perspectives converge on the same defects. A [CBR-vs-PBR comparison on design documents](https://www.researchgate.net/publication/3998247_An_experimental_comparison_of_checklist-based_reading_and_perspective-based_reading_for_UML_design_document_inspection) found similar effectiveness (PBR 69% vs CBR 70%) with PBR cheaper in time. Product translation: **multiple lenses/passes over the same diff (correctness pass, security pass, API-contract pass) is a validated inspection technique**, and it maps naturally onto multi-agent review.
- **Checklists: helpful, but weaker than hoped in the one rigorous modern test.** Wurzel Gonçalves, Fregnan, Baum, Schneider & Bacchelli, ["Do explicit review strategies improve code review performance? Towards understanding the role of cognitive load" (EMSE 2022)](https://link.springer.com/article/10.1007/s10664-022-10123-8) — professional-developer experiment, three treatments (ad hoc / checklist / guided checklist). From the abstract: *"the guided checklist is a more effective aid for a simple review, while the checklist supports reviewers' efficiency and effectiveness in a complex task. However, we did not identify a strong relationship between the guidance provided and code review performance."* The checklist can lower cognitive load, and (counterintuitively) higher load correlated with better performance in their mostly-novice sample. Related: throughout the experiment, **comprehension of the change was treated as the precondition for review performance**. Translation: don't ship a static checklist as the centerpiece; ship *comprehension support* with checklist-style coverage as scaffolding.
- **Review coverage predicts quality.** McIntosh, Kamei, Adams & Hassan, ["The Impact of Code Review Coverage and Code Review Participation on Software Quality" (MSR 2014, Qt/VTK/ITK)](https://dl.acm.org/doi/10.1145/2597073.2597076): components with **low review coverage suffer up to ~2 extra post-release defects, and low review *participation* (self-approval, no discussion, rushed reviews) up to ~5 extra**; reviews without discussion correlate with more defects. Industrial artifacts of coverage tracking: GitHub's per-file "Viewed" checkboxes, Gerrit's reviewed-file flags — but nothing mainstream tracks *hunk-level* coverage or time-on-hunk. That's an open product opportunity backed by the McIntosh result.
- **Eye-tracking corroborates partial reading.** Reviews of eye-tracking in SE ([SN Computer Science 2024 survey](https://link.springer.com/article/10.1007/s42979-024-03045-3), ["Eye movements in code review"](https://www.researchgate.net/publication/325479280_Eye_movements_in_code_review), [CodeGRITS toolkit](https://codegrits.github.io/CodeGRITS/static/paper.pdf)) show developers **fixate on a small fraction of lines, follow data-flow, and skip heavily** — experts skip more. Reviewers do not read diffs linearly or completely; a tool that *knows* what was actually attended to can close the loop.

### A4. Predicting where bugs hide in a diff

- **Just-in-time (change-level) defect prediction** is a mature field: Kamei et al.'s classic JIT models use churn, diffusion (files/dirs touched), history, and author experience. Fine-grained successors localize risk **within** the change: [Fine-Grained JIT Defect Prediction (JSS 2019)](https://www.sciencedirect.com/science/article/abs/pii/S0164121218302656) (file-level within commit), [JITLine (MSR 2021)](https://arxiv.org/pdf/2103.07068) (predicts defect-inducing commits *and* ranks defective lines), [hunk-level JIT prediction](https://www.researchgate.net/publication/352481799_Just-in-time_defect_prediction_for_software_hunks), [BARLineDP / line-level context models](https://www.researchgate.net/publication/355061269_The_Need_for_a_Fine-grained_approach_in_Just-in-Time_Defect_Prediction/fulltext/615c309d5a481543a87f379e/The-Need-for-a-Fine-grained-approach-in-Just-in-Time-Defect-Prediction.pdf), and LLM-era [CodeFlowLM incremental JIT prediction (arXiv 2512.00231)](https://arxiv.org/pdf/2512.00231). Robust signals: **churn size, scatter/diffusion, file hotspot history, complexity of surrounding code, author's familiarity with the touched code**.
- **Industrial risk heatmaps exist.** [CodeScene's Delta Analysis](https://docs.enterprise.codescene.io/versions/4.5.0/guides/delta/automated-delta-analyses.html) classifies **delivery risk per PR relative to the repo's own risk profile**, flags PRs touching top-ranked hotspots, and gates on Code Health decline — explicitly marketed as "prioritize your code reviews... focus your time where it's likely to be needed most." Devin Review's confidence-ranked bug flags (Part B) are the LLM-native version. Nobody yet fuses *statistical* JIT risk with *LLM semantic* risk into one per-hunk heat signal inside the reading flow — another open slot.
- **Caution from A2 applies:** risk models should be repo-relative and per-hunk, not global per-file-type priors.

### A5. Trust calibration / automation bias with LLM review (evidence for Part D)

- **The killer experiment:** Tufano, Martin-Lopez, Tayeb, Dabić, Haiduc & Bavota, ["Deep Learning-based Code Reviews: A Paradigm Shift or a Double-Edged Sword?" (arXiv 2411.11401)](https://arxiv.org/pdf/2411.11401) — controlled study, 29 expert reviewers, 50+ hours of monitored sessions. Reviewers given LLM-generated review comments as a starting point: **(1) anchoring — they focus on locations the LLM flagged and stop searching elsewhere; (2) they find more *low-severity* issues but NO additional *high-severity* bugs; (3) no time saved; (4) confidence unchanged.** This is the direct evidence that AI flags, presented naively, *narrow* human attention.
- **Automation complacency in SE is documented:** [Cognitive Biases in LLM-Assisted Software Development (arXiv 2601.08045)](https://arxiv.org/html/2601.08045v1) surveys over-trust/optimism toward LLMs causing unwanted actions; [Automated Code Review in Practice (arXiv 2412.18531)](https://arxiv.org/html/2412.18531v2) (industrial deployment study) records the exact failure mode: reviewers reasoning *"if any other issue existed, the bot would have written it."* General automation-bias background: [Springer AI & Society review of automation bias in human-AI collaboration](https://link.springer.com/article/10.1007/s00146-025-02422-7); the same effect is measured in [physician-LLM diagnostic reasoning](https://www.medrxiv.org/content/10.1101/2025.08.23.25334280.full.pdf) — it is a cross-domain human constant, not an SE quirk.
- Combined with A5's field-study finding that false positives erode trust in the other direction ([arXiv 2505.16339](https://arxiv.org/abs/2505.16339)), the design problem is *calibration*, in both directions.

## Part B — Devin deep-dive (Cognition, state mid-2026)

**Product surface.** [Devin Review](https://cognition.com/blog/devin-review) ([docs](https://docs.devin.ai/work-with-devin/devin-review), [app](https://app.devin.ai/review)) is Cognition's dedicated review product, positioned explicitly against the "Lazy LGTM problem" — "code review, not generation, is the bottleneck." Free during early release; works on any public/private GitHub PR (URL-swap, web app, or CLI); GitLab incl. Self-Managed supported; **no Bitbucket or Azure DevOps**; PAT connections are read-only (write actions need the GitHub App). Enterprise: ACU metering dashboards and per-PR auto-review spend caps.

**Features (names from docs):**
- **Smart diff organization** — the flagship: *"analyzes your code, groups together changes that are logically connected, orders the hunks of code, and explains each hunk"* — i.e., a generated guided walkthrough replacing the alphabetical file list. This is ClusterChanges/SmartCommit reborn with an LLM (validation for the whole category).
- **Copy and move detection** — renders moved code as moves, not delete+insert pairs.
- **Bug catcher** — flags ranked by **severity and confidence**: red = probable bugs, yellow = warnings, gray = FYI; "Severe" vs "Non-severe" labels; separate **flag classes**: *Investigate* (go verify yourself) vs *Informational* (explains behavior, no action).
- **Security scanning** — injection, auth flaws, secrets, SSRF, insecure deserialization, weak crypto; Critical/Warning severities with **CWE classifications**.
- **Codebase-aware chat** — ask questions from any hunk/comment/flag, answered with repo-wide context; **code changes from chat** can be pushed to the PR branch from within the review.
- **GitHub-compatible actions** — comment, approve, request changes, merge, close, auto-merge toggle, all synced to GitHub.
- **REVIEW.md** — per-repo review guidelines (conventions, security watchpoints) steering the auto-review; configurable triggers (auto / on-PR-creation / manual).

**DeepWiki** ([docs](https://docs.devin.ai/work-with-devin/deepwiki), [deepwiki.com](https://github.com/CognitionAI/deepwiki)) — Devin-generated, continuously updated wiki + architecture diagrams + conversational Q&A for any repo (swap `github.com` → `deepwiki.com`); free for open source; 30k+ repos indexed, ~4B LOC; exposes an MCP server (`ask_question`, `read_wiki_structure`, `read_wiki_contents`). It's the "repo comprehension" complement to Review, though the Review announcement does not describe deep integration between the two ([launch tweet](https://x.com/cognition_labs/status/1915816544480989288)).

**Assessment for a reviewer who wants to dig in themselves:**
- *Strengths:* the grouped, ordered, explained walkthrough is exactly what the Part A literature prescribes; confidence-tiered flags are a reasonable first stab at trust calibration; inline chat reduces context-switching (the top pain in [arXiv 2505.16339](https://arxiv.org/abs/2505.16339)).
- *Limitations:* it is a **web-app destination, not the reviewer's IDE** — no local checkout, no run-the-code loop, no jump-to-definition across the real workspace; it centers *Devin's findings* (flags list) over the *reviewer's own traversal*, which is precisely the anchoring pattern Tufano et al. warn about; no visible per-hunk coverage tracking of what the human actually read; no session pacing; DeepWiki context isn't woven into the review flow; ecosystem gaps (Bitbucket/ADO); and community reception notes flag-noise/verification burden ([HN thread on the launch](https://news.ycombinator.com/item?id=46711589)). Broader Devin-agent criticisms (over-engineering, getting lost in 100k+ LOC repos, "last 30%" incompleteness, subtle auth/crypto vulnerabilities) reinforce that its review output needs skeptical human traversal ([2026 field reviews](https://www.idlen.io/blog/devin-ai-engineer-review-limits-2026/), [deepfounder assessment](https://deepfounder.ai/devin-ai-in-2026-what-the-first-ai-software-engineer-actually-delivers/)).
- *Net:* Devin Review validates the market and the "guided walkthrough" mechanic, but it is **findings-first, not comprehension-first** — the reviewer-as-active-reader niche (coverage tracking, pacing, dig-in affordances, calibrated trust UI) remains open.

## Part C — The "Matt" lead

**Primary candidate (very high confidence): Matt Pocock — [`mattpocock/skills`](https://github.com/mattpocock/skills).** The TypeScript educator (Total TypeScript / AI Hero) open-sourced his personal `.claude` directory as a 21-skill collection — ["Skills for Real Engineers. Straight from my .claude directory."](https://github.com/mattpocock/skills/blob/main/README.md) It is **the most popular community skills repo in the Claude Code ecosystem (~48k stars as of April 2026, #2 on GitHub trending for six straight days)** ([guide](https://tosea.ai/blog/matt-pocock-skills-claude-code-guide), [walkthrough](https://note.com/shali_note/n/ne7849aa13605?hl=en)). Crucially it contains exactly the remembered pipeline — an idea→issues→implementation loop:
- **`grill-with-docs` / `grill-me`** — relentless interview to de-risk an idea before any code.
- **`to-spec`** — synthesize the conversation into a spec and **publish it to the issue tracker**.
- **`to-tickets`** — break a plan/spec into **tracer-bullet (vertical-slice) tickets with explicit blocking edges**.
- **`prd-to-issues`** — breaks a PRD into **independently-grabbable GitHub issues** via vertical-slice decomposition ([marketplace entry](https://claudemarketplaces.com/skills/mattpocock/skills/prd-to-issues)).
- **`implement`** — builds the work described by a spec or set of tickets, driving `/tdd` at pre-agreed seams; **`triage`** moves issues through a state machine; **`wayfinder`** plans multi-session work as investigation tickets.
- Install: `npx skills@latest add mattpocock/skills` or as a managed Claude Code plugin (`claude plugin install mattpocock-skills@mattpocock`), then `/setup-matt-pocock-skills` to bind the issue tracker. Philosophy: workflow enforcement (anti-misalignment, anti-verbosity, feedback loops) rather than capability extension.

**Secondary candidates (in case "Matt" was misremembered):**
- **Jesse Vincent (obra) — [Superpowers](https://github.com/obra/superpowers)** ([origin post](https://blog.fsck.com/2025/10/09/superpowers/)): brainstorm → written plan (2–5 min tasks with exact paths and verification) → subagent-per-task execution with two-stage review, TDD-gated. One of the fastest-growing OSS projects of 2026 ([Builder.io write-up](https://www.builder.io/blog/claude-code-superpowers-plugin)). It's plan-file-driven rather than GitHub-issues-driven.
- **Geoffrey Huntley — the ["Ralph Wiggum" loop](https://ghuntley.com/loop/)** ([interview](https://devinterrupted.substack.com/p/inventing-the-ralph-wiggum-loop-creator)): run the agent in a bare loop; each iteration starts with fresh context, reads project state from disk (PRD + progress file), completes one verified task, commits, exits. Packaged variants: [snarktank/ralph](https://github.com/snarktank/ralph) (autonomous loop until all PRD items complete), a [Ralph PRD→GitHub-issues converter skill](https://mcpmarket.com/tools/skills/github-prd-to-issues-converter) (parent/sub-issue hierarchy, dependency mapping via GraphQL), and [AI Hero's Ralph workshop — run by Matt Pocock](https://www.aihero.dev/events/turn-ai-agents-into-autonomous-software-engineers-with-ralph), which may be the exact source of the "Matt + idea→pipeline" memory.
- Honorable mentions: [automazeio/ccpm](https://github.com/automazeio/ccpm)-style PRD→epic→GitHub-issues→parallel-agents systems, [aaddrick/claude-pipeline](https://github.com/aaddrick/claude-pipeline) (multi-agent pipeline with `handle-issues` batch orchestration and an end-to-end `implement-issue` skill), and curated lists ([ComposioHQ/awesome-claude-skills](https://github.com/ComposioHQ/awesome-claude-skills)).

**Recommendation for building THIS product:** adopt the pattern, modestly. (1) Install `mattpocock/skills` and use `grill-with-docs` → `to-spec` → `prd-to-issues`/`to-tickets` to turn the existing vision/requirements docs into vertical-slice GitHub issues with blocking edges — the tracer-bullet decomposition fits a solo-plus-agents project well and keeps every slice demoable. (2) Use `implement`/`triage` (or plain Claude Code sessions per issue) rather than a fully autonomous Ralph loop at this stage: Ralph shines on well-specified greenfield grind, but this product's core UX is novel and needs human taste in the loop — exactly the lesson of Part A/D. (3) Dogfood loop: every agent-built PR from that pipeline becomes test input for the review tool itself.

## Part D — Ten research-backed design principles for a reviewer-first AI review tool

1. **Order the diff for comprehension, never alphabetically.** Position bias is measured: earlier files get more attention (+23% comments from smart reordering; only 10% of devs endorse alphabetical). Generate an intent-ordered walkthrough. ([Fregnan/Bagirov](https://arxiv.org/abs/2306.06956), [FSE 2024](https://arxiv.org/abs/2404.10703), [ICSE 2026](https://conf.researchr.org/details/icse-2026/icse-2026-research-track/310/Breaking-the-Alphabet-Rethinking-File-Ordering-in-Code-Review))
2. **Decompose tangled changes into intent-coherent chunks.** >40% of real changesets are decomposable; decomposition measurably improves understanding. Use LLM untangling with def-use grounding. ([ClusterChanges](https://www.microsoft.com/en-us/research/wp-content/uploads/2016/02/barnett2015hdh.pdf), [di Biase 2019](https://pmc.ncbi.nlm.nih.gov/articles/PMC7924728/), [SmartCommit](https://dl.acm.org/doi/abs/10.1145/3468264.3468551))
3. **Pace the session: ~200–400 effective LOC per sitting, ≤60–90 minutes, then force a break.** Defect detection collapses past ~400 LOC / 500 LOC-per-hour. Budget the walkthrough into sittings and show remaining effort. ([SmartBear/Cisco](https://static0.smartbear.co/support/media/resources/cc/book/code-review-cisco-case-study.pdf))
4. **Track review coverage at hunk level and surface it.** Unreviewed and undiscussed code ships more post-release defects (up to +2 / +5 per component); GitHub's "Viewed" checkbox is the crude ancestor — make attention coverage a first-class artifact. ([McIntosh MSR 2014](https://dl.acm.org/doi/10.1145/2597073.2597076))
5. **Answer information needs in place: rationale, alternatives, context per hunk.** Reviewers' dominant unmet needs are "why", "is this the right approach", and cross-file context — not more findings. Inline, codebase-aware Q&A beats a flags list. ([Pascarella CSCW 2018](https://dl.acm.org/doi/10.1145/3274404), [Bacchelli & Bird 2013](https://sback.it/publications/icse2013.pdf), [arXiv 2402.09528](https://arxiv.org/html/2402.09528))
6. **Design against anchoring: AI flags must widen attention, not narrow it.** With LLM pre-reviews, experts stop searching beyond flagged locations, catch no extra high-severity bugs, and save no time. Separate "guide/comprehension" mode from "findings" mode; reveal flags per-chunk *after* the reviewer has read the chunk, or on demand. ([Tufano et al.](https://arxiv.org/pdf/2411.11401))
7. **Calibrate trust explicitly, in both directions.** Show confidence tiers (probable / warning / FYI, à la Devin), state what the AI did *not* check, and never imply completeness — "the bot would have said something" is the documented failure mode; false-positive floods are the opposite one. ([arXiv 2412.18531](https://arxiv.org/html/2412.18531v2), [arXiv 2505.16339](https://arxiv.org/abs/2505.16339), [Devin Review docs](https://docs.devin.ai/work-with-devin/devin-review))
8. **Concentrate reviewer attention with per-hunk risk, computed repo-relatively.** Fuse JIT-defect-prediction signals (churn, diffusion, hotspot history, author familiarity) with LLM semantic risk into a heat signal that drives ordering and altitude; distrust file-type priors (language↔defect effects are tiny). ([JITLine](https://arxiv.org/pdf/2103.07068), [CodeScene delta analysis](https://docs.enterprise.codescene.io/versions/4.5.0/guides/delta/automated-delta-analyses.html), [Berger TOPLAS 2019](https://dl.acm.org/doi/10.1145/3340571))
9. **Vary reading altitude, but verify before compressing.** Collapse verifiably-mechanical content (generated files, moves, mechanical renames — cf. `linguist-generated`, Devin's move detection) and summarize "diluted" hunks — but only after per-hunk checks (no logic, no security surface), because low-attention categories (e.g., test code) demonstrably hide issues. ([GitHub docs](https://docs.github.com/en/repositories/working-with-files/managing-files/customizing-how-changed-files-appear-on-github), [Spadini ICSE 2019](https://sback.it/publications/icse2019a.pdf))
10. **Offer multiple review lenses, and keep the human the accountable gatekeeper.** Perspective-based passes (correctness, security, API-contract) achieve broader defect coverage than one generic pass; checklists alone don't move performance — comprehension does. Structure the product as human-controlled checkpoints over agent stages, not an auto-approve pipeline — especially since most agent PRs currently get *no* human review at all. ([Basili PBR](https://www.cs.umd.edu/~mvz/handouts/emp_pbr.pdf), [Wurzel Gonçalves EMSE 2022](https://link.springer.com/article/10.1007/s10664-022-10123-8), [arXiv 2605.17548](https://arxiv.org/pdf/2605.17548), [arXiv 2605.02273](https://arxiv.org/pdf/2605.02273))
