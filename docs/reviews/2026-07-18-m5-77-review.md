# Review — #77 (consumer-first / chapter mode + AI augmentation), 2026-07-18

Post-merge review of #77 (the default flip to consumer-first/tests-before/chapter mode and its AI
augmentation), done before the M5 UI slices (#78–#80) build on chapter mode. Read this before
touching the chapter-order or overlay-v2 code.

## Verdict

**Sound to build the M5 UI on — no must-fix blocker.** The overlay-v2 data path, partition
validation, the `checkOrder` pre-gate, fail-open everywhere, and the section-id coupling guards
are all correct, and the UI consumes the already-applied `aiBook` (right whether the overlay
applied or fell back to tier-0).

## Fixed in this pass

- **Chapter-order size guard measured the wrong text (correctness).** The guard tripped on
  `manifest.estimatedTokens`, which core computed from the *raw-id* `renderChunkOrderManifest`, but
  the model is sent the *aliased* manifest (`c1…cN`, ~3–4× shorter). So AI ordering — #77's default
  — silently refused on moderate/large diffs (≈80 chunks → raw estimate ~10k > 8000 guard, real
  aliased prompt ~2.8k) and fell back to tier-0 exactly where narrative ordering matters most. The
  small eval subjects slipped under the raw limit, which is why the 9/0 verdict didn't expose it.
  Fix: the guard now measures the actual aliased prompt in `runChapterOrderJob`; regression test in
  `order-job.test.ts` ("the size guard measures the aliased prompt").
- **Root-cause dedup.** The two near-identical renderers (`renderChunkOrderManifest` raw /
  `renderAliasedManifest`) were the reason the guard could drift from the sent text. Unified into
  one `renderChunkOrderManifest(manifest, labelOf)` parameterized by an id→label fn; the server
  passes the alias map. Server output is byte-identical, so `CHAPTER_ORDER_PROMPT_VERSION` is
  unchanged and the eval stays valid. `estimatedTokens` dropped from `ChunkOrderManifest`.
- **`assembleSections` O(occurrences × chunks) label lookup (nit).** Replaced the per-occurrence
  `gitOrder.find` with a `fileById` map built once (`chapters.ts`).

## Deferred (tracked here, not urgent)

- **`/api/book` and `export.md?order=ai` recompile the chapter book twice per request** via
  `applyChapterOverlay`, which internally compiles tier-0 (freshness check) and then again with the
  composition — redundant with the compile `getBook` already ran. Recompute-per-request is a
  deliberate no-stale-cache choice, but the freshness half is avoidable: the server already holds
  `built.book` + `built.storyComposition`, so it could freshness-check inline and compile once.
  Not fixed now: no measurement shows it's slow at real sizes, and the fix changes the
  `applyChapterOverlay` contract (accept a precompiled tier-0) — worth a measurement first. The
  "(milliseconds)" perf assertion in the server comment is unverified; treat it as such.

## Verified clean (don't re-audit)

- **Overlay v2 apply/validate**: `applyChapterOverlay` fails open to `undefined` before touching the
  composition; `validateChapterComposition` is a true partition check (missing/foreign/dup/empty all
  named).
- **Chapter order round-trip**: alias↔id map is bijective; the parser rejects unknown/dup/missing
  aliases and re-keys rationales to `chapter:<anchorChunkId>`; the `checkOrder` pre-gate runs before
  persisting.
- **checkOrder chunk-position semantics**: direction and test-placement gates invert correctly with
  config; cyclic pairs go to non-gating `cycleInversions`; `weaveTests` satisfies the test gate by
  construction.
- **Section-id coupling trap**: `isFileModeBook` + the throws in `sectionAnchors` / `testImplAnchors`
  guard the file-path assumption; every #77 `buildChunkGraph` caller passes the file-mode book. The
  trap the prior review named is closed.
- **Daemon branching**: `resolveChunkGraph` fails open to an empty graph; `bookCache` carries the
  matching graph/composition/input; `aiBook` is set only when a fresh v2 overlay applies.
- **Web v2 handling**: `overlay.version === 2` returns `aiBook ?? book`; the AI-order indicator stays
  reference-based; rationale keys and cross-file `label` render correctly.
