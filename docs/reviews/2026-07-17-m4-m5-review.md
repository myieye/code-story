Scope: everything landed 2026-07-17 in one build window — `4cd9b34..origin/main` (PRs
#81–#90). That's AI-order default-on (#71), the M4 context surface (#64–#67), and the M5
graph + chapter linearizer (#74–#76). ~30 candidates examined; the survivors below are ranked
fix-now / fix-soon / deliberately-not-fixed. **No code changed in this pass — review only.**

Read against spec 04 (`docs/spec/04-context-payloads.md`) and spec 05
(`docs/spec/05-chunk-graph-and-traversals.md`), and matched to the discipline of the two
prior reviews (`2026-07-16-architecture-review.md`, `2026-07-16-m2-review.md`).

Headline: the parallel-agent work is unusually clean — fail-open discipline holds on every
overlay loader I checked, R-050 kind-layering is real (the `default`-less switch is genuinely
banned and `edgesOfKinds` is used), R-036 register on the new user-facing strings is fine, and
`resolveTsSpecifier` was correctly lifted to shared core. The real debt is **seam drift between
the three resolvers built the same day** and **lifecycle triplication in an
already-doubled-in-size server.ts/cli.ts** — exactly the "made in three places" risk the review
was asked to hunt. One latent precision gap rises above the rest because the edge it produces
now *drives ordering*, not just a display affordance.

## Fix now

- **The chunk-graph `calls` resolver never got the #91 import-justification its sibling
  resolver has — and its edges now steer the story order.**
  `context-resolve.ts:120-133` (`resolveChanged`) only considers a changed file as a definition
  candidate when it is the chunk's own file *or* is import-connected to it (`cf.path === chunk.file || imported.has(cf.path)`)
  — this is the #91 fix ("8 `CreateEntry` definitions, one in the diff — confidently resolved
  to the wrong class"). The graph builder's `resolveTarget` (`chunk-graph-build.ts:132-138`)
  did **not** get that rule: when a name is defined in exactly one changed file it is chosen
  outright (`files.length === 1`), with **no** requirement that the caller import it. So the
  exact false-positive class #91 was filed to kill still exists on the `calls` path — and
  since #75/#76, those `calls` edges feed `buildSpine` (the chapter order), `checkOrder`'s
  chunk-position gate, and the upcoming neighbor strip. A wrong edge is now a wrong *chapter
  placement* or a phantom inversion, not just a wrong jump affordance (spec 05: "a wrong edge is
  worse than a missing one"). Slice-0's audit (0.976 self-graded) didn't surface it only because
  none of the three subjects happened to have a lone-in-diff name the caller doesn't import.
  *Failure scenario*: a diff touches one function `foo` and, separately, a caller `bar` that
  calls a *different, unchanged* `foo` — the graph draws `bar → foo(changed)`, and consumer-first
  chapters seat `bar` immediately before the wrong `foo`. *Fix shape*: lift the shared
  "resolve a referenced name to its defining changed symbol, justified by an import edge" step
  into one function both `resolveChanged` and `resolveTarget` call (they already share the same
  ambiguity philosophy and both use a first-match symbol walk); the divergence is the bug. Add
  the missing test (see test-debt below) — the current suite only covers the *two*-file
  ambiguity case (`chunk-graph-build.test.ts:69`), which passes under both rules and so hides
  the divergence.

- **The bulk context job aborts wholesale on a single chunk's resolver throw — a fail-open
  claim the code doesn't keep.** `runContextJob` (`context-job.ts:60-75`) does
  `const payload = await deps.resolve(chunk)` with no per-chunk guard; an exception propagates
  to the job's outer `catch` (`server.ts:562`, `cli.ts:321`) and marks the *entire* fill
  `failed`. Spec 04 frames facts as best-effort per chunk ("unresolved references are simply
  absent"), and the on-demand `GET /api/context` (`server.ts:492`) is equally unguarded — a
  throw there is a 500, not the `{ payload: null }` the `ContextResponse` doc promises
  ("resolution never throws into the book flow", `api.ts`). Today the throw probability is low
  (`extractSymbols`/`extractReferences` return `undefined`/`[]` rather than throw for
  unsupported input, and `fileAt` is already `.catch`-wrapped at the server seam), which is why
  this is a small fix rather than an outage — but the guarantee is asserted in three doc
  comments and enforced in none. *Fix shape*: wrap the per-chunk `resolve` in the job loop so one
  failure records an empty payload and continues, and wrap the GET resolve to return
  `{ payload: null }` on throw. Cheap; makes the stated contract true.

## Fix soon

- **Three resolvers, two of them silently diverging.** There are now three symbol-resolution
  passes: `import-graph.ts` (file→file), `context-resolve.ts` (name→definition, changed +
  unchanged), and `chunk-graph-build.ts` (name→defining chunk). The good half: `resolveTsSpecifier`
  is shared (`import-graph.ts` ↔ `context-resolve.ts`). The unshared half is the changed-file
  name→symbol disambiguation, which exists twice (`resolveChanged` vs `resolveTarget`) with
  *different* rules — the #91 gap above is one symptom, and the known arity-blind overload miss
  (2379-e06 in the slice-0 baseline; `findSymbol`/`flattenSpans` both keep the first same-name
  span) is a second weakness that will have to be fixed in *both* places. Separately, all three
  now re-`extractSymbols` head content that `computeChunks` already parsed and threw away — the
  spec-04 "no retained parse" tradeoff, but it is three passes now, not two. *Fix shape*: extract
  the changed-file resolver as one core-adjacent helper with an explicit documented contract
  (candidates = own-file ∪ imported; unique-after-justification wins; ambiguity → none), and let
  both consumers adapt its output (definition body vs chunk id). Consider threading the chunker's
  symbol tables through so the second/third parse disappears.

- **Job lifecycle is triplicated in `server.ts`, which nearly doubled today (373→593 lines).**
  The three job GETs carry a byte-identical orphan re-read (`server.ts:280-286` order,
  `356-362` narration, `507-513` context — the "still `running` with no live handle after one
  re-read = orphan" dance). The three POSTs carry the same live-guard + `try/finally` +
  status-record-write lifecycle; order alone was extracted (`kickOrderJob`), narration and
  context are inlined copies. This is precisely the seam the m2-review moved `claude-cli.ts` out
  on. *Fix shape*: a small `jobRuntime<Record>({ live, liveRecord, reload, orphanError })`
  helper owning the live-handle + orphan-resolution, and a `runManagedJob(record, work)` wrapper
  owning the running→done/failed persistence. Three call sites collapse to three thin configs.
  Do it now while there are exactly three and the shape is proven — a fourth job type will make
  it a four-way copy-paste.

- **`cli.ts` grew the same way (259→498 lines) as a flat if-chain, each branch re-running the
  pipeline.** `--dump-chunks`, `--dump-graph`, `--dump-chunk-graph`, `--check-order`,
  `--dump-manifest`, `--context`, `--dump-context`, and the export block each independently call
  `diffRange` + `computeChunks` + `compileBook` (`cli.ts:162-352`). The context-fill `persist`
  closure (`cli.ts:302-311`) is a *fourth* variant of the store-cap-then-write logic that
  `server.ts:448-464`'s `persistPayload` also implements. *Fix shape*: a `withCompiled(range)`
  helper returning `{ files, chunks, contents, graph, compiled }` once, and share the
  cap-and-write persister with the server. The m2-review already flagged CLI flag-precedence as
  a deferred subcommand-refactor; this is the same file asking again, louder.

- **Chapter mode is wired only through the CLI; nothing here surfaces it to the daemon — state
  this loudly so no one assumes the web honors config.** `getBook()` (`server.ts:144-153`)
  always calls `compileBook` (file mode); `compileChapterBook`, `buildChunkGraph`, and
  `.code-story.json` config are reachable only via `--export`/`--check-order` (`cli.ts:214-247`,
  `431-475`). This matches spec 05's sequencing (UI slices 5–6 gated on slice 0; #77 flips the
  default later), so it is not a bug — but it is a sharp coupling trap: `sectionAnchors`,
  `fileImportEdges`, and `testImplAnchors` all key on `section.id === <file path>`
  (`chunk-graph.ts:90-98`, `book.ts:247`). A chapter book's section ids are `chapter:<chunkId>`,
  so the moment someone wires `compileChapterBook` into the graph/anchor path those helpers
  silently produce empty anchor maps. Worth an explicit note in the code (or an assertion that
  the book passed to `sectionAnchors` is file-mode) before slice 5 lands.

- **Two full Tarjan/SCC implementations now coexist and must be kept in lockstep by hand.**
  `book.ts:201` (`stronglyConnected` over files, used by `cycleBreakIndex`) and
  `chapters.ts:181` (`cycleBreakSeed`'s inline Tarjan over chunks) encode the same
  "break the git-earliest member of a condensation-source SCC" policy. They will drift: a fix to
  the cycle-break tie-break (the kind #32/#52 already made once at file grain) now has to be made
  twice. *Fix shape*: one generic `sourceSccToBreak(nodes, successors, rank)` in a shared module,
  parameterized on the node type. Low urgency (both are correct and tested today), but it is a
  standing double-maintenance cost.

- **`d`-key on an unfetched payload expands the panel with no focus move and no announcement.**
  `toggleDefinitionsFor` (`BookPage.tsx:189-201`) only moves focus / announces on the synchronous
  `'expanded'` outcome. When the payload isn't cached yet, `context.toggle` returns `'none'`,
  records `wantExpand`, and the panel later auto-expands on fetch arrival (`useContextPanels.ts:36`)
  — but focus stays on the chunk and `say(...)` never fires, so a keyboard/screen-reader user
  gets a silent, un-focused panel. In practice `ensureFetched` runs on cursor focus so the cache
  is usually warm, making this a race, not a constant. *Fix shape*: have the deferred-expand path
  in the hook signal back (a callback or a returned promise) so the page can focus + announce on
  arrival, mirroring the synchronous path.

## Deliberately not fixed (with reasons)

- **Panel expand-state and `panelEls` focus map are keyed by `chunkId`, not occurrence key.**
  `BookPage.tsx:422-427` keys `panelExpanded` and the `registerPanelEl` map on `chunk.id`. This
  is *correct* for the payload itself (facts are shared across occurrences by design, spec 04),
  and file-mode / chapter-mode both emit exactly one occurrence per chunk today — so there is no
  live bug. It only bites once a chunk has two occurrences (neighbor-strip jumps, slice 5):
  expanding one would expand both, and `focus()` would target whichever DOM node registered last.
  The architecture review's occurrence-vs-chunk note applies here; the right time to key the
  panel DOM map by `occurrenceKey` (keeping the payload lookup by chunk id) is when slice 5
  introduces the second occurrence, not speculatively now.

- **On-demand `GET /api/context` resolves low-signal stubs and leftover chunks; the bulk job
  excludes them.** Spec 04's non-goal ("no payloads for low-signal stubs or the leftovers
  section") is enforced by `eligibleContextChunks` (`context-job.ts:8-23`) for bulk, but the GET
  (`server.ts:484`) resolves any chunk id in the book. Harmless — a stub resolves to a near-empty
  payload and the web only fetches on cursor focus — and guarding it would duplicate the
  eligibility predicate at the GET. Leave it; revisit only if a stub ever resolves something
  noisy enough to clutter the affordance.

- **`checkChunkOrder` mixes idioms for kind selection** — `edgesOfKinds(CALLS_DFS_KINDS)` for
  the calls pass (`check-order.ts:110,122`) but an inline `edge.kind !== 'exercises'` /
  `=== 'exercises'` for the test pass (`check-order.ts:141-144`). Not an R-050 violation (no
  exhaustive switch, additive kinds still pass through), just two idioms for one concept. Cosmetic;
  fold into an `EXERCISES_KINDS` set when the exercises pass next changes.

- **Slice-0 edge-precision gate is still half-open** (Tim's blind labels uncommitted, per the
  dogfood-0 baseline "Slice 0" section). Not code — it is the recorded human-read gate, and the
  UI slices (5–6) are correctly blocked on it. Flagged only so the fix-now precision gap above is
  read as "harden before the blind audit is trusted," not "after."

- **K=3 subjects, thin `calls` layer.** The slice-0 sample is 42 calls edges (1/30/11), leaning
  almost entirely on PR 2357. Recorded in the baseline doc as a "fourth calls-heavy subject would
  help." Dogfood work, not a code finding.
