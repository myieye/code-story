* The ultimate tool should consider how old and large the PR is. I.e. more generally it should have a keen awareness of how to most wisely suggest/apply changes vs not iterating forever, because it's very valuable to get the PR merge
* Other related aspects are: how critical the change is (e.g. a dev-feature, a quick vibe-code of a prototype we want to get out, something really critical etc.)
* I expect the resulting story/narration to help me think of those aspects and help my find the best way to get a PR to a responsibly mergeable state rather than perfecting every detail. Nitpicks are sometimes good, sometimes just a real pain. (not just nitpicks: refactors, optimizations etc.)
* Initial attempts to get AI to generate a story for me were rather weak. It's really critical that the AI breaks hunks up into truly manageable pieces. I want to think as little as possible and just be able to read a thorough story of the code
* It's critical that I can make manual changes directly where the code is shown (either directly to the code or if easier, a git patch could be generated - hopefully via script - and applied (also then living in the history of the review/thread))
* This is important: I'm constantly overwhelmed with how dense AI generated explanations are. The easiest text format for me to read is an exciting novel. Code is not a novel, but it's incredibly important that the story/narration reads really well. Avoid using overly complex terminology and language. It doesn't have to be cave man language, but maybe think high-school english. Obviously devs are reviewing it, so...I think you know what I'm trying to say. Not too dense, light, easily accessible etc.
